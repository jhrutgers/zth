1 # Using Zth on Bare-Metal ARM
2 
3 There is preliminary support for bare-metal ARM with Zth.
4 Most of Zth's source code is just C++, so usable on any target.
5 Only the context switching and hardware-related stuff like `clock_gettime()` is to be ported.
6 
7 The directory `examples/arm` contains a set of files that shows how to do this.
8 This examples targets a Qemu virt machine to run the examples.
9 This virt machine uses an ARM Cortex-A15 with VFP (without NEON).
10 The implementation is not very nice, but shows you where to start, if you have your own ARM system.
11 The compiler, linker script, and startup files are probably supplied to you by the manufacturer of your board.
12 
13 To get the stuff running, do this:
14 
15 1. Create an out-of-tree build directory, just like normal:
16 
17  mkdir build-arm
18  cd build-arm
19  cmake ..
20 
21 2. Now, cross-compile all examples in one go.
22  The files will be generated in the directory `examples/arm/zth-arm/examples`.
23 
24  cd build-arm
25  make arm
26 
27  That was easy!
28  Ok, I was cheating a bit. This actually happened (see also `examples/arm/CMakeLists.txt`):
29 
30  1. Download and extract ARM toolchain (`arm-none-eabi-gcc` and friends).
31  The toolchain is installed in `examples/arm/gcc-arm-none...`
32 
33  make arm-none-eabi-gcc
34 
35  2. Download, extract and build `newlib`.
36  This uses the toolchain, as installed in the previous step.
37  `newlib` is compiled specifically for our target processor.
38 
39  make arm-newlib
40 
41  3. Build BSP (`libarm-bsp.a`).
42  This library contains `crt0.c` and `startup.S`, which both reside in `examples/arm`.
43  These files include the interrupt vector for the ARM, intialization of the VFP, C/C++-runtime boot code, and the system calls that `newlib` needs, like `_read()` and `_write()`.
44  All binaries compiled for our target need this BSP library further on.
45 
46  make arm-bsp
47 
48  4. A new CMake out-of-tree build is initiated with the propert toolchain configuration.
49  Basically, this calls CMake with `-DCMAKE_TOOLCHAIN_FILE=examples/arm/toolchain.cmake` in the directory `examples/arm/zth-arm`.
50 
51  make zth-arm
52 
53  5. Build Zth for ARM. This target includes everything above, but also runs `make` in `examples/arm/zth-arm`.
54  As a result, all examples are cross-compiled for ARM in `examples/arm/zth-arm/examples`.
55  The examples are built using the `toolchain.cmake` configuration, as provided above, and are linked against our `newlib` and `libarm-bsp.a`.
56 
57  make arm
58 
59 3. Run Qemu with one of the examples.
60  I have tested it with Qemu 3.0.0.
61  (Note: Press 'Ctrl-A X' to terminate Qemu)
62 
63  qemu-system-arm -machine virt -cpu cortex-a15 -s -m 8M -nographic -kernel examples/arm/zth-arm/examples/1_helloworld
64 
65  If you have a debug build, the output will include the banner, saying something like:
66 
67  > zth::banner: Zth 0.1.0 g++-8.3.1 C++14 baremetal newlib3.1.0 arm debug sjlj
68 
69  Note that there is no OS that supplies `poll()`. So reading stdin is a bit tricky and not all examples behave the same way as at your PC.
70  Moreover, 0MQ isn't available either, of course.
71 
72 4. Connect `gdb` to Qemu. If you supply the `-S` argument to Qemu, it waits for `gdb` to connect before it starts.
73  That might make debugging a bit easier.
74 
75  examples/arm/gcc-arm-none-eabi-8-2019-q3-update/bin/arm-none-eabi-gdb -ex 'target remote :1234' -ex 'break main' -ex 'layout split' examples/arm/zth-arm/examples/1_helloworld
